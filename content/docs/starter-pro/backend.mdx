## Backend foundations that make the product real

Starter Free gives you a **frontend SaaS surface**.

Starter Pro gives you the **backend structure, data model, and server-side foundations** required to run a real product.

This is where your app stops behaving like a demo and starts behaving like a system:

- real persistence
- real users
- real ownership
- real billing state
- real server-side rules
- real product behavior

### What this solves

A lot of SaaS products do not fail because the UI is weak.

They fail because the backend becomes unclear:

- data ownership is duplicated
- auth and billing are not connected cleanly
- access rules leak into components
- product logic is scattered across pages
- schema decisions become expensive to change
- mocked data turns into fragile production logic

Starter Pro gives you a structured foundation so backend work can scale with the product instead of slowing it down.

## Architecture

### Data model responsibilities

A real SaaS backend needs models that represent the business layer, not only UI data.

Starter Pro is designed around responsibilities such as:

| Area | Purpose |
| --- | --- |
| User | Identity, profile, ownership, sessions |
| Account | OAuth provider connections |
| Session | Active authenticated sessions |
| Customer | Billing relationship to Stripe |
| Product | Sellable product or plan family |
| Price | Stripe price and commercial plan |
| Subscription | Access, status, lifecycle, and renewals |
| Invoice | Payment history and billing state |
| WebhookEvent | Reliable event processing |
| Domain models | Your actual product entities |

<Callout type="info" title="Why this matters">
  A SaaS backend is not only CRUD.
  It is the place where identity, revenue, ownership, and product behavior meet.
</Callout>

## Implementation path

<Steps>
  <Step>
    ### Validate the existing foundation

    Before adding your domain logic, confirm that auth, billing, database setup, and local seed flows work.
  </Step>

  <Step>
    ### Define your first domain model

    Add the product entity that matters most to your SaaS.

```prisma title="schema.prisma"
model Project {
  id        String   @id @default(cuid())
  name      String
  ownerId   String
  createdAt DateTime @default(now())
  updatedAt DateTime @updatedAt
}
```
  </Step>

  <Step>
    ### Add ownership rules

    Connect product data to the authenticated user, customer, or organization model.

```ts title="server/projects/access.ts"
export function canAccessProject(projectOwnerId: string, userId: string) {
  return projectOwnerId === userId;
}
```
  </Step>

  <Step>
    ### Keep product logic server-side

    The UI should call the product flow.

    It should not own sensitive rules.
  </Step>

  <Step>
    ### Connect billing when needed

    If the feature is paid or plan-dependent, add plan-aware checks near the server boundary.
  </Step>
</Steps>

## What you avoid building yourself

Without Starter Pro, you usually need to design and wire:

- database schema foundations
- auth-to-user relationships
- billing-to-customer relationships
- subscription access rules
- protected route behavior
- webhook state persistence
- product ownership checks
- delivery access rules
- domain separation patterns

This is often time-consuming, easy to duplicate, and expensive to refactor later.

### Backend rules

Use these rules when extending the backend.

### Production checklist

Before treating the backend as launch-ready, validate:

- database migrations
- seed data
- user model
- session model
- billing customer mapping
- subscription state
- webhook event handling
- ownership checks
- protected data access
- plan-aware access logic
- server-side validation
- error handling
- deployment environment variables

<Callout type="info" title="Validation principle">
  Do not build new product complexity on top of an unvalidated backend.
  Confirm the foundation first.
</Callout>

## Common questions

<Accordions type="single">
  <Accordion title="Is Starter Pro a full backend product?">
    No. It gives you the backend foundations needed to launch faster. You still build your domain models, workflows, and product-specific rules.
  </Accordion>

  <Accordion title="Can I change the schema?">
    Yes. The schema is meant to be extended. Keep auth, billing, and ownership boundaries clear when adding product-specific models.
  </Accordion>

  <Accordion title="Should access control live in the frontend?">
    No. The frontend can reflect access state, but sensitive checks should live on the server or in backend-aware logic.
  </Accordion>

  <Accordion title="How does billing connect to backend logic?">
    Billing state should be synchronized from Stripe and stored in your app so server-side logic can decide access reliably.
  </Accordion>
</Accordions>
