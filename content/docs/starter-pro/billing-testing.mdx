## Why billing testing matters

Billing systems are distributed state systems.

A successful redirect does not guarantee synchronized subscription state.

The frontend can look correct while the backend state is still wrong:

- checkout may complete
- the user may return to the app
- the subscription may not be synchronized
- premium access may remain locked
- webhook processing may fail silently
- billing state may drift from Stripe

That is why billing must be tested as a lifecycle, not as a button.

```txt title="billing-lifecycle.txt"
Checkout
  ↓
Stripe
  ↓
Webhook verification
  ↓
Subscription sync
  ↓
Application billing state
  ↓
Plan-aware access
  ↓
Protected product routes
```

<Callout type="info" title="Production insight">
  The most dangerous billing bugs often appear after checkout succeeds.
  Validate the state synchronization path, not only the payment page.
</Callout>

## Stripe CLI

Use Stripe CLI to forward local webhook events to Starter Pro.

```bash title="stripe-cli.sh"
stripe listen --forward-to localhost:3000/api/webhooks/stripe
```

Then configure the webhook secret in `.env.local`.

```bash title=".env.local"
STRIPE_WEBHOOK_SECRET=whsec_xxxxx
```

<Callout title="Webhook rule">
  If the webhook secret is wrong, billing sync should fail safely.
  Do not bypass webhook verification during development.
</Callout>

## Common questions

<Accordions type="single">
  <Accordion title="Why are webhooks required?">
    Because redirects alone cannot confirm what actually happened inside Stripe. Webhooks are the backend synchronization layer.
  </Accordion>

  <Accordion title="Should access checks happen client-side only?">
    No. Sensitive billing-aware access should remain server-verifiable. The UI can reflect access state, but the server should protect valuable resources.
  </Accordion>

  <Accordion title="Do I need to test failed payments before launch?">
    Yes. Failed payment and cancellation states are part of real subscription lifecycle behavior and should not be discovered after launch.
  </Accordion>
</Accordions>
