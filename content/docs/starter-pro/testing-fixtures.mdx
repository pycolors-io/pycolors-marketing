## Production-shaped local testing

Starter Pro includes seeded account states designed to validate realistic SaaS behavior during development.

The goal is not only to create users.

The goal is to simulate:

- authentication states
- onboarding states
- billing states
- protected route behavior
- subscription-aware product access

## Why this exists

Most SaaS products become difficult to validate locally because developers must manually recreate account edge cases repeatedly.

Starter Pro reduces that friction by exposing reusable testing states.

```txt title="testing-fixtures-model.txt"
Seed fixtures
  ↓
Account states
  ↓
Billing states
  ↓
Protected routes
  ↓
Realistic product validation
```

## Why account states matter

Real SaaS products behave differently depending on:

- verification lifecycle
- billing status
- provider state
- onboarding state
- protected access level

Without realistic fixtures, developers often validate only happy paths.

That creates hidden production risk later.

<Callout type="info" title="Production insight">
  A single happy-path user is not enough to validate a SaaS product.
  Real products require multiple identity, billing, and access states.
</Callout>

## Enable fixtures

```bash title=".env.local"
SEED_AUTH_FIXTURES=true
```

Then run:

```bash title="seed.sh"
pnpm app:setup
```

## Recommended validation flow

<Steps>
  <Step>
    ### Enable fixture seeding

    Set `SEED_AUTH_FIXTURES=true` in your local environment.
  </Step>

  <Step>
    ### Run setup

```bash
pnpm app:setup
```
  </Step>

  <Step>
    ### Validate authentication states

    Test verified, unverified, password-based, and OAuth-oriented account behavior.
  </Step>

  <Step>
    ### Validate billing-aware states

    Confirm that premium, free, and upgrade-gated surfaces behave as expected.
  </Step>

  <Step>
    ### Validate mobile and PWA behavior

    Confirm that the same account states remain usable in mobile and standalone app contexts.
  </Step>
</Steps>

## Common questions

<Accordions type="single">
  <Accordion title="Why include multiple account states?">
    Because real SaaS products behave differently depending on verification, billing, provider state, and onboarding lifecycle.
  </Accordion>

  <Accordion title="Should fixtures exist in production?">
    No. Fixtures are intended only for local development and controlled staging environments.
  </Accordion>

  <Accordion title="Why not create users manually?">
    Manual setup is slower, inconsistent, and easy to forget. Fixtures make core states repeatable across local environments.
  </Accordion>
</Accordions>
