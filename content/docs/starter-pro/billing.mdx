## Billing lifecycle

<Steps>
  <Step>
    ### User starts upgrade

    The user clicks an upgrade CTA from pricing, billing, or a gated feature.

```txt title="entry-points.txt"
/pricing
/billing
/protected-feature
```
  </Step>

  <Step>
    ### Backend creates Checkout session

    The browser should not create Stripe sessions directly.
    Your backend creates the Checkout session using server-side Stripe keys.

```ts title="server/billing/create-checkout-session.ts"
const session = await stripe.checkout.sessions.create({
  customer: stripeCustomerId,
  mode: "subscription",
  line_items: [{ price: priceId, quantity: 1 }],
  success_url: `${appUrl}/billing?success=true`,
  cancel_url: `${appUrl}/pricing?canceled=true`,
});
```
  </Step>

  <Step>
    ### Stripe confirms payment

    Stripe handles payment, subscription creation, and customer billing operations.

    The UI should not assume success only because a redirect happened.
  </Step>

  <Step>
    ### Webhook updates app state

    Webhooks are the synchronization layer.
    They verify what actually happened in Stripe and update your database.

```txt title="webhook-events.txt"
checkout.session.completed
customer.subscription.created
customer.subscription.updated
customer.subscription.deleted
invoice.payment_succeeded
invoice.payment_failed
```
  </Step>

  <Step>
    ### App reflects billing state

    Product access, billing page, invoice history, and plan-aware UI should read from synchronized backend state.
  </Step>
</Steps>

<Callout title="Production rule">
  Redirects are UX. Webhooks are truth.
</Callout>

## Architecture

### Data model responsibilities

Billing needs enough structure to represent customer and subscription state inside your app.

Typical responsibilities include:

| Area | Purpose |
| --- | --- |
| Customer | Maps app users or organizations to Stripe customers |
| Product | Represents sellable SaaS products |
| Price | Represents Stripe prices and commercial plans |
| Subscription | Tracks current subscription state |
| Invoice | Stores invoice references and payment status |
| WebhookEvent | Tracks processed Stripe events for reliability |

<Callout type="info" title="Why this matters">
  Stripe is the payment source.
  Your app still needs synchronized billing state to protect access and render product UI correctly.
</Callout>

### Plan gating

Billing becomes valuable when it controls product behavior.

Starter Pro gives you the foundation to build logic such as:

- free vs paid access
- plan-based feature flags
- subscription status checks
- billing-aware settings
- upgrade prompts
- cancellation behavior
- failed payment messaging

```ts title="plan-access.ts"
export function canAccessProFeature(subscriptionStatus: string) {
  return subscriptionStatus === "active" || subscriptionStatus === "trialing";
}
```

Keep plan logic clear, centralized, and server-verifiable.

### Implementation checklist

Before treating billing as launch-ready, validate:

- checkout session creation
- checkout success flow
- checkout cancel flow
- customer mapping
- billing portal session creation
- webhook signature verification
- subscription created event
- subscription updated event
- subscription canceled event
- invoice payment succeeded event
- invoice payment failed event
- plan-aware access logic
- billing page state
- user-facing billing error copy

<Callout type="info" title="Validation principle">
  Do not trust billing until you validate the full lifecycle:
  checkout, webhook, database sync, and product access.
</Callout>

## Common questions

<Accordions type="single">
  <Accordion title="Is this only a pricing page?">
    No. Starter Pro billing is about the revenue layer: Checkout, portal, webhooks, subscription state, invoices, and plan-aware access.
  </Accordion>

  <Accordion title="Can I start with one-time payments instead of subscriptions?">
    Yes, but the billing model should still keep a clear customer, payment, and access boundary. Keep the product decision separate from the Stripe integration pattern.
  </Accordion>

  <Accordion title="Should the frontend decide if a user has access?">
    No. The UI can reflect access state, but sensitive access checks should be server-verifiable and based on synchronized billing state.
  </Accordion>

  <Accordion title="Why are webhooks required?">
    Redirects are not enough. Webhooks tell your application what actually happened inside Stripe after payment, renewal, cancellation, or failure.
  </Accordion>
</Accordions>
