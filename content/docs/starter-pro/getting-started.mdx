## Recommended setup order

Follow this order if your goal is speed.

### 1. Install dependencies

Start by installing the workspace dependencies.

### 2. Configure local environment

Set the core variables first:

- app URL
- database URL
- auth secret
- optional OAuth providers
- optional Stripe variables
- optional email variables

Do not overconfigure too early.

Get the foundation running first.

### 3. Run database setup

Generate the database client, run migrations, and seed local fixtures.

This helps you validate the auth and billing layers faster.

### 4. Start the app

Run the local development server and confirm that the baseline is working.

### 5. Validate the foundations

Before changing product logic, confirm that the important layers behave correctly:

- sign in
- registration
- protected routes
- seeded users
- billing surfaces
- settings pages

### 6. Replace branding and product copy

Only after the baseline works should you start replacing:

- product name
- logo
- metadata
- copy
- navigation labels
- billing text
- email sender identity

### 7. Build your real product inside the shell

Now the important part starts:

- your workflows
- your domain entities
- your product logic
- your differentiating features

---

## What to avoid

The most common mistake is trying to redesign everything before the foundations are validated.

Avoid this order:

1. rewrite the whole UI
2. rename everything
3. add features immediately
4. ignore auth and billing validation
5. discover infrastructure issues later

That path feels productive, but it usually delays launch.

A better rule:

> validate the base first, then customize, then build the product.
