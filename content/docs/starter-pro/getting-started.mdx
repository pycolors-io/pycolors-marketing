## Get from purchase to product work faster

Starter Pro is built to shorten the path between:

- buying the product
- running it locally
- validating the foundations
- customizing the product
- shipping your own SaaS faster

The goal is not to study the codebase for days.

The goal is to get the production baseline running, confirm the important systems, then start building your product on top of it.

<Callout title="After purchase">
  Download and unzip Starter Pro from your claim email before running the setup commands below. Need the email again? Use [purchase recovery](/orders/recover) to resend your access link.
</Callout>

<Callout title="Core setup principle">
  Validate the foundation first.
  Customize second.
  Build your product third.
</Callout>

## What this page helps you do

By the end of this page, you should understand:

- what to configure first
- what to validate before editing deeply
- how to avoid wasting time on low-leverage setup
- how to move from installation to real product work
- when to focus on auth, billing, backend, and branding

### The fastest serious path

Starter Pro gives you a production-ready baseline with:

- authentication
- billing
- protected routes
- settings
- dashboard structure
- backend-ready foundations
- secure delivery path

You are not starting from a blank codebase.

You are starting from a product-shaped system that is already much closer to launch.

<Callout type="info" title="Important">
  The fastest path is not rewriting everything.

  The fastest path is validating what already exists, then replacing only what belongs to your product.
</Callout>

## Setup flow

<Steps>
  <Step>
    ### Download and unzip Starter Pro

    Open your claim email and use the access link to download your package.
    Unzip it before running the setup commands below.
  </Step>

  <Step>
    ### Install dependencies

```bash title="terminal"
pnpm install
```

    Start by installing the workspace dependencies.

    This gives you the local baseline required to run the app,
    generate the database client,
    and validate the foundation.
  </Step>

  <Step>
    ### Configure local environment

    Start with the core variables first.

```bash title=".env.local"
NEXT_PUBLIC_APP_URL=http://localhost:3000
DATABASE_URL=
AUTH_SECRET=
```

    Add optional variables only when you are ready to validate those systems.

```bash title=".env.local"
AUTH_GITHUB_ID=
AUTH_GITHUB_SECRET=
AUTH_GOOGLE_ID=
AUTH_GOOGLE_SECRET=

STRIPE_SECRET_KEY=
STRIPE_WEBHOOK_SECRET=
STRIPE_PRICE_STARTER_PRO_ONE_TIME=

RESEND_API_KEY=
AUTH_EMAIL_FROM=
```

    Do not overconfigure too early.

    Get the baseline running first.
  </Step>

  <Step>
    ### Run database setup

```bash title="terminal"
pnpm db:generate
pnpm db:migrate
pnpm db:seed
```

    Generate the database client,
    run migrations,
    and seed local fixtures.

    This helps you validate auth,
    users,
    billing surfaces,
    and local development flows faster.
  </Step>

  <Step>
    ### Start the app

```bash title="terminal"
pnpm dev
```

```txt title="browser"
http://localhost:3000
```

    Confirm that the app loads and the main product routes are reachable.
  </Step>

  <Step>
    ### Validate foundations

    Before changing product logic, confirm the important layers:

    - sign in
    - registration
    - protected routes
    - seeded users
    - billing surfaces
    - settings pages
    - dashboard shell

    This gives you confidence that the baseline works before you customize it.
  </Step>

  <Step>
    ### Replace branding and product copy

    Once the baseline is validated, update:

    - product name
    - logo
    - metadata
    - navigation labels
    - billing copy
    - email sender identity
    - marketing links

    Branding comes after validation, not before.
  </Step>

  <Step>
    ### Build your product layer

    Now add your real product:

    - domain entities
    - workflows
    - onboarding
    - dashboards
    - customer-facing features
    - internal operations

    Keep auth, billing, and protected access intact while you build your differentiation.
  </Step>
</Steps>

### What to avoid

The most common mistake is trying to redesign everything before the foundations are validated.

Avoid this order:

1. rewrite the whole UI
2. rename everything
3. add features immediately
4. ignore auth and billing validation
5. discover infrastructure issues later

That path feels productive, but it usually delays launch.

A better rule:

> validate the base first, then customize, then build the product.

### Fastest path to value

If your goal is speed, focus on this sequence:

1. get the app running
2. validate auth
3. validate billing
4. validate protected routes
5. replace branding
6. add your first domain entity
7. ship the first vertical slice

This gives you the best speed-to-confidence ratio.

<Callout title="High-leverage setup">
  Starter Pro pays for itself when you avoid rebuilding the foundation and start building what customers actually care about.
</Callout>

### First product changes

After setup, the best first changes are usually:

- replace branding
- update metadata
- define your core domain model
- add your first real entity
- connect the first real workflow
- keep auth and billing intact

Examples:

- projects
- workspaces
- reports
- AI generations
- CRM records
- internal operations data

The exact feature does not matter.

What matters is building your product inside the existing structure instead of fighting it.
