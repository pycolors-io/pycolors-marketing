| `public/sw.js` | lightweight service worker |
| `public/pwa/*` | icons and screenshots |

<Callout title="Architecture rule">
  Keep the PWA layer explicit and easy to audit. A starter product should teach the buyer how the system works, not hide it behind unnecessary magic.
</Callout>

## Online-first boundaries

| Area | Recommended approach |
| --- | --- |
| Authentication | Online-first |
| Billing | Online-first |
| Admin actions | Server-validated |
| Account settings | Server-validated |
| Subscription state | Backend source of truth |
| Offline support | Fallback-only baseline |

<Callout title="Production rule">
  Redirects, cached UI, and offline pages are UX.
  Server state remains the source of truth for authentication, billing, permissions, and subscriptions.
</Callout>

## Common questions

<Accordions type="single">
  <Accordion title="Is Starter Pro a full offline-first SaaS?">
    No. Starter Pro includes offline-ready foundations and fallback routing.
    It does not pretend that billing, authentication, permissions, or admin data can safely work offline without a dedicated synchronization system.
  </Accordion>

  <Accordion title="Why not cache the dashboard aggressively?">
    Dashboards often contain user-specific, billing-aware, or permission-sensitive data.
    Caching those surfaces without a clear invalidation strategy can show stale or incorrect information.
  </Accordion>

  <Accordion title="Should I use next-pwa or Workbox?">
    You can add a more advanced PWA library later.
    Starter Pro starts with an explicit lightweight baseline because it is safer, easier to learn, and better suited to auth and billing-heavy SaaS products.
  </Accordion>

  <Accordion title="Is the PWA layer required for the app to work?">
    No. The app should work normally without service worker support.
    PWA behavior is progressive enhancement.
  </Accordion>
</Accordions>
