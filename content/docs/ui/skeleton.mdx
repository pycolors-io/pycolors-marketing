## Preserve layout while content loads

The **Skeleton** component is a visual placeholder used during loading states.

It helps:

- preserve layout structure
- reduce layout shift
- improve perceived performance
- communicate that content is expected soon
- keep product surfaces stable while data loads

Skeleton is a **pure UI primitive**. It renders shapes and does not own any loading, fetching, or async logic.

<Callout title="Core idea">
  Skeleton shows the shape of content that is coming. It is not a message, an explanation, or an error state.
</Callout>

### Import

```tsx title="skeleton-import.tsx"
import { Skeleton, type SkeletonProps } from "@pycolors/ui";
```

### Usage patterns

<Steps>
  <Step>
    #### Match the final content shape

    Skeleton should approximate the content it replaces: text lines, avatars, cards, or rows.
  </Step>

  <Step>
    #### Preserve layout height

    Keep the loading state close to the final height to reduce layout shift.
  </Step>

  <Step>
    #### Keep it simple

    Prefer a few clear shapes over a detailed fake version of the final UI.
  </Step>

  <Step>
    #### Replace quickly when data is ready

    Skeleton is temporary. It should disappear as soon as real content can render.
  </Step>

  <Step>
    #### Use another state when loading ends

    Empty, error, and success states need their own UI. Skeleton only represents loading.
  </Step>
</Steps>

## Skeleton vs Spinner vs EmptyState vs Alert

| Situation | Use |
| --- | --- |
| Layout is known and content is loading | Skeleton |
| Short unknown background task | Spinner or inline loading |
| Request succeeded but no data exists | EmptyState |
| Request failed | Alert |
| Long-running operation requiring explanation | Dedicated status UI |
| Blocking app-level loading | Route-level loading UI |

<Callout type="info" title="Decision rule">
  Use Skeleton for expected content. Use EmptyState for valid absence. Use Alert for failure.
</Callout>

| Prop | Type | Default | Description |

### Accessibility

Skeleton is visual only.

Guidelines:

- Skeleton always renders with `aria-hidden="true"` — this is a fixed guarantee of the component and cannot be overridden by a passed-through prop
- Skeleton should never receive focus
- Skeleton should not be used as a status message
- Do not announce every placeholder to screen readers
- Provide accessible loading semantics at a higher level when needed
- The pulse animation respects `prefers-reduced-motion` and is disabled automatically for users who request it
- Replace skeletons with real content as soon as possible

<Callout title="Accessibility rule">
  Skeletons describe visual shape for sighted users. They are not content and should not be announced as content.
</Callout>

## Common questions

<Accordions type="single">
  <Accordion title="Should Skeleton include text for screen readers?">
    Usually no. Skeleton should be hidden from assistive technologies. If loading status matters, announce it at the parent region level.
  </Accordion>

  <Accordion title="Should I use Skeleton or a spinner?">
    Use Skeleton when the final layout is known. Use a spinner or inline loading state when the shape is unknown or the action is small.
  </Accordion>

  <Accordion title="Can Skeleton be used in tables?">
    Yes. Use skeleton cells inside table rows to preserve table structure while data loads.
  </Accordion>

  <Accordion title="How detailed should a skeleton be?">
    Keep it simple. Match the main structure, not every detail.
  </Accordion>
</Accordions>
