# Build interfaces that scale with your product

PyColors UI components are **primitives**, not finished widgets.

They are designed to:

- adapt to your product
- stay maintainable over time
- work inside real SaaS applications

This is not a UI kit for demos.
This is a foundation for **products that evolve**.

---

## Where this fits

PyColors UI is part of a larger system:

- **Starter Free** → validate product UI
- **UI components** → build consistent interfaces
- **Starter Pro** → wire auth, billing, backend

👉 UI is the layer between **idea** and **real product**

---

## The mental model

- **Tokens define the system**
- **Components stay small**
- **Patterns handle complexity**

If a component becomes complex:

👉 it probably belongs in **Patterns**, not in primitives

---

## Component categories

### Surfaces

Structure your interface and group content.

- Card
- Alert

---

### Actions

Trigger user intent.

- Button

---

### Inputs

Collect and validate user data.

- Input

---

### Status & metadata

Display state and labels.

- Badge

---

## Composition rules

Keep your UI scalable:

- prefer composition over configuration
- use semantic tokens (`bg-card`, `text-muted-foreground`)
- avoid one-off variants
- keep components small and predictable
- one interaction per component

These rules prevent UI drift as your product grows.

---

## When to use Patterns instead

Use Patterns when you need:

- full flows (auth, settings, pricing)
- async states
- validation logic
- real product sections

👉 Components = building blocks
👉 Patterns = real product usage

## Why this matters

Most UI systems break when the product grows:

- too many variants
- inconsistent styles
- duplicated components

PyColors avoids this by keeping primitives **simple and composable**.
