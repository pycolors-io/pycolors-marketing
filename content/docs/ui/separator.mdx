### Visual separation without visual noise

The **Separator** component creates a thin divider between related pieces of interface.

Use it for:

- separating page sections
- grouping card content
- dividing toolbar actions
- separating metadata rows
- creating vertical dividers in compact layouts
- improving scanability without adding heavy containers

Separator is intentionally minimal. It uses PyColors semantic tokens such as `bg-border`, so it adapts to light mode, dark mode, and product theme overrides.

<Callout title="Core idea">
  Separator should clarify structure, not decorate the page.
</Callout>

### Import

```tsx title="separator-import.tsx"
import { Separator } from "@pycolors/ui";
```

## Decorative vs semantic

By default, Separator is decorative and uses `role="none"`.

Set `decorative={false}` when the separator communicates a meaningful boundary for assistive technologies.

### Usage patterns

<Steps>
  <Step>
    #### Start with spacing

    Use spacing first. Add Separator only when visual grouping is still unclear.
  </Step>

  <Step>
    #### Keep dividers subtle

    Use semantic border tokens. Avoid strong colors unless the divider is part of a very specific branded surface.
  </Step>

  <Step>
    #### Pick the right orientation

    Use horizontal dividers for stacked content and vertical dividers for compact rows.
  </Step>

  <Step>
    #### Avoid over-fragmenting the page

    Too many separators make the UI feel heavy and harder to scan.
  </Step>
</Steps>

## Separator vs Border vs Card

| Situation | Use |
| --- | --- |
| Split content inside the same surface | Separator |
| Define the edge of a surface | Border |
| Group content into a distinct object | Card |
| Divide toolbar action groups | Separator |
| Separate unrelated page regions | Section spacing or Card |
| Emphasize a critical state | Alert or semantic color |

<Callout title="Decision rule">
  Use Separator inside a surface. Use Card when the content deserves its own surface.
</Callout>

## API

| Prop | Type | Default | Description |
| --- | --- | --- | --- |
| `orientation` | `"horizontal" \| "vertical"` | `"horizontal"` | Divider direction |
| `decorative` | `boolean` | `true` | Removes separator semantics when true |
| `className` | `string` | — | Additional Tailwind classes |
| `...props` | `React.HTMLAttributes<HTMLDivElement>` | — | Standard div attributes |

### TypeScript types

```ts title="separator-types.ts"
export interface SeparatorProps
  extends React.HTMLAttributes<HTMLDivElement> {
  orientation?: "horizontal" | "vertical";
  decorative?: boolean;
}
```

### Accessibility

- Decorative separators use `role="none"`.
- Semantic separators use `role="separator"`.
- Semantic separators include `aria-orientation`.
- Most visual dividers should be decorative.
- Do not use Separator as a replacement for headings.
- Do not rely on a divider alone to explain hierarchy.

<Callout title="Accessibility rule">
  If the separator does not add meaning for screen reader users, keep it decorative.
</Callout>

## Common questions

<Accordions type="single">
  <Accordion title="Should Separator be semantic by default?">
    No. Most separators are visual only. PyColors keeps them decorative by default to avoid unnecessary screen reader noise.
  </Accordion>

  <Accordion title="Why does vertical Separator sometimes not appear?">
    Vertical separators need height from the parent layout. Use them inside a row with a fixed height, `items-stretch`, or an explicit height class.
  </Accordion>

  <Accordion title="Should I use Separator between every list item?">
    Usually no. Use spacing, row backgrounds, or borders only when density requires stronger grouping.
  </Accordion>

  <Accordion title="Can Separator use custom colors?">
    Yes through `className`, but prefer semantic tokens like `bg-border`, `bg-border-subtle`, or theme-specific tokens when available.
  </Accordion>
</Accordions>
