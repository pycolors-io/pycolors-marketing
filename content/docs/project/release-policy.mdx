# How PyColors ships

PyColors is designed to move fast **without becoming unstable**.

To achieve that, releases are structured around **two layers**:

- **Packages** → precise, technical versioning
- **Ecosystem** → clear, human-readable progress

---

## The model

### 1. Packages → Semantic Versioning

All public packages (like `@pycolors/ui`) follow **SemVer**.

```txt
MAJOR → breaking changes
MINOR → new features (backward compatible)
PATCH → fixes and small improvements
```

Examples:

```
1.0.3 → bug fixes
1.1.0 → new components
2.0.0 → breaking changes
```

### Source of truth

- GitHub Releases
- CHANGELOG.md (generated by Changesets)

---

### 2. Ecosystem → Weekly releases

The website communicates a weekly release cadence.

This includes:

- UI improvements
- Starter updates (Free → Pro)
- Templates and product layers

**Packages ship when ready.
The ecosystem ships every week.**

---

## Why this approach

Most projects struggle with one of two problems:

- releases are too slow → no momentum
- releases are too chaotic → no trust

PyColors balances both:

- fast iteration at the package level
- consistent communication at the product level

---

## What this guarantees

- predictable updates
- no hidden breaking changes
- continuous progress
- clear separation between technical truth and product narrative

---

## Non-negotiables

These rules keep the system consistent:

- Development happens in the core monorepo
- Public repos are distribution surfaces
- All packages use Changesets
- GitHub Releases are the technical truth
- The website reflects product progress

---

## Changesets workflow

All packages follow the same release process:

1. Make changes in the monorepo
2. Add a changeset (patch / minor / major)
3. Merge to main
4. CI updates version + changelog
5. CI publishes to npm + GitHub Releases

---

## Where to track updates

| Surface | Purpose |
|--------|--------|
| /changelog | Weekly ecosystem updates |
| GitHub Releases | Technical release details |
| CHANGELOG.md | Package-level changes |

---

## What is public

This release model is public on purpose.

It helps:

- developers understand how updates work
- users trust the product direction
- contributors align with the system

Internal operational details (CI, secrets, infra) are not exposed.

---

## Mental model

**Packages move fast
The ecosystem stays predictable**
