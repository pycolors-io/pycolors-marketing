# Build a layout that can survive product growth

Most UI libraries stop at components.

Real SaaS products need structure:

- navigation
- header
- content areas
- responsive behavior
- clear interaction boundaries

This guide explains how to compose PyColors UI primitives into a **real product layout** without locking your app into a rigid shell.

---

## The real problem

A SaaS layout is not just a sidebar plus a header.

It needs to support:

- product navigation
- contextual actions
- dense data views
- settings flows
- desktop and mobile behavior
- growth over time

A layout that works for week one often breaks by month three.

That is why layout should be treated as a **composition problem**, not a component problem.

---

## PyColors approach

PyColors gives you:

- primitives
- tokens
- product guidance

It intentionally does **not** force:

- a mandatory app shell
- fixed navigation APIs
- framework-specific layout rules
- global layout abstractions you do not own

That keeps your product flexible.

---

## Mental model

> Layout is not a component.
> Layout is a composition of responsibilities.

A simple conceptual model looks like this:

```tsx
<AppShell>
  <AppSidebar />
  <Main>
    <AppHeader />
    <Content />
  </Main>
</AppShell>
```

This is not a required API.

It is a useful separation of concerns:

- navigation
- context
- content

---

## Core primitives used in SaaS layouts

A real product layout can be built from a small set of primitives:

- Button → actions and nav triggers
- Sheet → mobile navigation
- Dropdown Menu → user menu and row actions
- Tabs → settings sections
- Table → data-heavy content
- Dialog → confirmations and modal flows

This is the key idea:

You do not need “layout magic”.
You need reliable primitives and good composition.

---

## Sidebar strategy

### Desktop

On larger screens:

- sidebar stays visible
- navigation is persistent
- product areas remain easy to scan
- tables and long views use full height

### Mobile

On smaller screens:

- sidebar is hidden
- navigation moves into a Sheet
- header exposes a menu trigger
- content gets priority

This pattern is predictable, scalable, and accessible.

---

## Header responsibilities

A SaaS header usually carries:

- page title
- contextual description
- primary action
- user/account menu
- mobile nav trigger

Use primitives for each responsibility:

- Button for actions
- Dropdown Menu for account controls
- Sheet trigger for mobile nav

Keep headers light.

Headers should provide context, not become control panels.

---

## Content areas

Most product pages fall into two groups.

### Data-heavy views

- tables
- filters
- row actions
- pagination
- bulk actions

### Configuration views

- settings forms
- tabs
- confirmations
- dialogs
- account flows

A strong SaaS layout supports both without forcing a different shell for each.

---

## Example: projects page

A realistic data page might combine:

- Table for the list
- Badge for status
- Dropdown Menu for row actions
- Dialog for destructive confirmation
- Sheet for mobile navigation

That is how a layout starts supporting real product operations.

---

## Example: settings page

A realistic settings page often combines:

- Tabs to separate concerns
- Input for editable fields
- Button for actions
- Dialog for risky confirmations

The layout stays stable while the content changes.

That is what good product structure looks like.

---

## Why this approach scales

Composition-based layouts scale better because they let you:

- evolve navigation independently
- swap sections without rewriting the shell
- adapt different product areas
- support more teams later
- keep layout ownership inside the app

This is more useful than shipping a rigid “app shell” that becomes expensive to bend.

---

## What PyColors intentionally avoids

PyColors does not ship:

- a mandatory app shell
- routing-specific layout logic
- layout state you do not control
- forced product structure

Those decisions belong to your product.

The UI system should support them, not own them.

---

## When to create your own AppShell

Create a custom AppShell when:

- navigation becomes complex
- multiple product areas need consistency
- several people are working in the same product
- layout drift starts appearing

That is the right point to formalize your layout.

Not before.

---

## Practical rule

Start with composition.

Introduce your own AppShell only when the repetition becomes real.

That gives you:

- less premature abstraction
- more product control
- better long-term structure

---

## Summary

- SaaS layouts are compositions, not components
- primitives should stay small
- layout responsibilities should stay separate
- PyColors provides guidance, not rigid structure

Build the layout deliberately.
Keep the system flexible.
