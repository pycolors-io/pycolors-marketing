# Build forms that behave predictably

Forms are one of the most fragile parts of a product.

They combine:

- user input
- validation rules
- async submission
- accessibility requirements
- error handling

This guide explains how to build forms with validation using PyColors UI primitives without hiding logic inside opaque abstractions.

---

## Why forms fail so often

Forms look simple on the surface.

In practice, they have to coordinate:

- field values
- validation feedback
- loading state
- submission success or failure
- accessibility and semantics

This is why good form systems are not only about inputs.

They are about **clear responsibilities**.

---

## Core principles

Before any tool choice, a few principles matter more than APIs.

### 1. Forms are systems, not components

A form is not a single reusable component.

It is a composition of:

- inputs
- buttons
- validation rules
- submission logic
- feedback states

Trying to abstract the entire form too early usually creates:

- too many props
- hidden rules
- poor debuggability
- hard-to-reuse abstractions

In PyColors, forms are documented as **patterns**, not primitives.

---

### 2. Validation belongs to the data layer

UI components should:

- display values
- reflect errors
- show state clearly

They should **not define what is valid**.

Validation rules belong close to:

- your schema
- your domain model
- your form logic

The UI consumes the result.

---

### 3. Error states should be explicit

Good forms make failure understandable.

That means:

- short field-level error messages
- clear global feedback for submission errors
- no hidden failure state
- no reliance on color alone

---

## Recommended minimal stack

This guide uses a minimal, production-friendly setup:

- **React Hook Form** for form state
- **Zod** for schema validation
- **PyColors UI** for rendering

You can swap tools later if needed.

The principles remain the same.

---

## Mental model

> Inputs display state.
> Schemas define truth.
> Forms coordinate the flow.

Keeping those responsibilities separate makes forms easier to:

- reason about
- debug
- extend over time

---

## Example: basic form with validation

We will build a simple form with:

- required fields
- inline validation errors
- disabled submit while loading
- global error feedback

---

## Schema

Validation rules come first.

```ts
import { z } from "zod";

export const formSchema = z.object({
  email: z.string().email("Invalid email address"),
  password: z.string().min(8, "Password must be at least 8 characters"),
});
```

This keeps validation explicit and close to the data model.

---

## Form setup

```ts
import { useForm } from "react-hook-form";
import { zodResolver } from "@hookform/resolvers/zod";

const form = useForm({
  resolver: zodResolver(formSchema),
  defaultValues: {
    email: "",
    password: "",
  },
});
```

The form state now handles:

- field values
- validation output
- submission state

---

## Rendering inputs

PyColors Input reflects validation state.

It does not own validation.

```tsx
<form onSubmit={form.handleSubmit(onSubmit)} className="space-y-4">
  <Input
    label="Email"
    type="email"
    {...form.register("email")}
    error={form.formState.errors.email?.message}
  />

  <Input
    label="Password"
    type="password"
    {...form.register("password")}
    error={form.formState.errors.password?.message}
  />
</form>
```

This works well because:

- errors remain explicit
- inputs stay focused on rendering
- validation logic stays outside the primitive

---

## Submit button states

Buttons should reflect form state, not invent it.

```tsx
<Button
  type="submit"
  disabled={form.formState.isSubmitting}
>
  {form.formState.isSubmitting ? "Saving…" : "Save"}
</Button>
```

Practical rules:

- disable submit while submitting
- avoid double submission
- use clear loading text
- do not rely only on animation to show progress

---

## Global error feedback

Some failures are not tied to one field:

- network failure
- permission issues
- unexpected server errors

Use Alert for those cases.

```tsx
{formStateError && (
  <Alert variant="destructive">
    <AlertTitle>Submission failed</AlertTitle>
    <AlertDescription>
      Please try again later.
    </AlertDescription>
  </Alert>
)}
```

Use field-level errors for field problems.
Use a global error block for form-level problems.

---

## Full example

```tsx
function ExampleForm() {
  const form = useForm({
    resolver: zodResolver(formSchema),
  });

  async function onSubmit(values) {
    try {
      await save(values);
    } catch {
      setError("Something went wrong");
    }
  }

  return (
    <Card>
      <CardHeader>Account settings</CardHeader>
      <CardContent>
        <form
          onSubmit={form.handleSubmit(onSubmit)}
          className="space-y-4"
        >
          <Input
            label="Email"
            {...form.register("email")}
            error={form.formState.errors.email?.message}
          />

          <Input
            label="Password"
            type="password"
            {...form.register("password")}
            error={form.formState.errors.password?.message}
          />

          <Button
            type="submit"
            disabled={form.formState.isSubmitting}
          >
            Save changes
          </Button>
        </form>
      </CardContent>
    </Card>
  );
}
```

## UX rules

These rules make forms feel more stable in production.

### Validation

- validate at meaningful moments
- keep messages short and specific
- keep field-level feedback close to the field
- make the reason for failure obvious

### Submission

- disable submit while loading
- avoid duplicate submissions
- keep button text clear
- show form-level failure where needed

### Accessibility

- use visible labels
- do not rely only on color for errors
- keep feedback text readable
- preserve semantic form structure

- evolve
