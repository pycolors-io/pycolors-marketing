# Async states define product quality

Modern products spend a large part of their life outside the happy path.

Data loads.
Lists are empty.
Requests fail.

How you handle these moments directly affects:

- perceived performance
- user confidence
- product clarity
- overall product quality

This guide explains how to design **loading**, **empty**, and **error** states with PyColors UI primitives.

---

## Why this matters

A product is not judged only by its successful states.

It is also judged by how it behaves when:

- data is still loading
- nothing exists yet
- a request fails
- the user needs guidance

Well-designed async states make the product feel:

- stable
- intentional
- trustworthy

Poor async states make even good products feel unfinished.

---

## The three async states

Every data-driven interface moves through three distinct states:

1. **Loading** → data is being fetched
2. **Empty** → request succeeded, but there is nothing to show
3. **Error** → request or action failed

These states are not interchangeable.

Each has:

- a different meaning
- a different UX goal
- a different visual response

---

## Mental model

> **Skeleton = shape**
> **EmptyState = explanation**
> **Alert = problem**

Keeping these roles separate is one of the easiest ways to make a product feel more professional.

---

## Loading states → Skeleton

### When to use Skeleton

Use **Skeleton** when:

- data is loading
- the layout is already known
- content is expected soon

Skeleton preserves structure and reduces visual jank.

It helps users understand:

> “Something is coming here.”

### What Skeleton is for

Skeleton is:

- a placeholder
- a layout-preserving state
- a temporary visual structure

Skeleton is **not**:

- content
- product feedback
- an explanation

## Empty states → EmptyState

### When to use EmptyState

Use **EmptyState** when:

- the request succeeded
- the result is valid
- there is nothing to display yet

This is not a failure.

This is a real, expected product state.

An EmptyState should answer two questions:

1. **Why is this empty?**
2. **What can the user do next?**

## Error states → Alert

### When to use Alert

Use **Alert** when:

- a request fails
- an action fails
- the user needs immediate feedback
- the product cannot proceed as expected

An error state communicates a **problem**.

It should be:

- clear
- calm
- actionable

---

## Choosing the right state

| Situation | Component |
|---|---|
| Data is loading | Skeleton |
| Request succeeded, but there is no data | EmptyState |
| Request or action failed | Alert |

Never blur these roles.

A product feels clearer when each state has one job.

---

## UX rules

These rules make async states feel more stable in production.

### Loading

- preserve the layout whenever possible
- use Skeleton when the structure is known
- avoid replacing the whole page with a spinner
- remove loading placeholders quickly when data is ready

### Empty

- explain why the section is empty
- offer a useful next step when relevant
- do not treat empty as failure
- keep the tone neutral and helpful

### Error

- say what happened
- tell the user what they can do next
- avoid vague system language
- use inline feedback for blocking problems
