# Radius controls how the system feels

Corner radius has a strong effect on interface feel.

It influences whether the product appears:

- sharp
- soft
- dense
- calm
- premium
- utilitarian

PyColors defines a small radius scale so corners stay consistent across the system.

---

## Base radius

PyColors starts from one base variable.

```css
:root {
  --radius: 0.625rem;
}
```

Derived tokens are generated from that base:

```css
@theme inline {
  --radius-sm: calc(var(--radius) - 4px);
  --radius-md: calc(var(--radius) - 2px);
  --radius-lg: var(--radius);
  --radius-xl: calc(var(--radius) + 4px);
}
```

That makes it possible to adjust the global feel of the product from one place.

---

## Radius scale

| Token | Typical use |
|------|-------------|
| --radius-sm | compact controls, pills, dense UI |
| --radius-md | default interactive elements |
| --radius-lg | cards and surfaces |
| --radius-xl | modals and standout containers |

---

## Tailwind mapping

```ts
borderRadius: {
  sm: "var(--radius-sm)",
  DEFAULT: "var(--radius-md)",
  md: "var(--radius-md)",
  lg: "var(--radius-lg)",
  xl: "var(--radius-xl)",
}
```

---

## Usage guidance

- use rounded-md as the default for most controls
- use rounded-lg for cards and main surfaces
- use rounded-xl sparingly for standout layers
- use rounded-sm in denser UI contexts
- avoid arbitrary radius values inside components

If a new radius keeps repeating, introduce a token instead of inventing local values.

---

## Why this matters

Radius seems small, but it is one of the fastest ways for systems to drift.

A stable radius scale keeps the product feeling intentional even as more components appear.
