# Tokens make the system coherent

Design tokens are the single source of truth for PyColors UI’s visual language.

They encode the atomic decisions that every component consumes.

That is what makes the system:

- consistent
- predictable
- easier to evolve globally

Tokens are not components.
They are not raw stylesheets.
They are design decisions expressed as variables.

---

## What tokens are for

A token represents meaning, not a one-off visual choice.

In practice, tokens answer questions like:

- what radius should this system use?
- how much elevation should a card have?
- how should surfaces feel visually related?

Instead of solving these decisions inside each component, the design system solves them once.

---

## System flow

```text
Design decisions
  ↓
Tokens (CSS variables)
  ↓
Tailwind utilities
  ↓
UI Components
  ↓
Guides and product surfaces
```

This is why components can stay small while the system still feels expressive.

---

## Current token categories

### Radius

Corner geometry used across:

- buttons
- inputs
- cards
- overlays

### Shadows

Elevation used across:

- controls
- cards
- menus
- modals

PyColors intentionally starts with a small token surface.

This keeps the system understandable and avoids token sprawl too early.

---

## Why tokens matter

Using tokens gives you:

- global consistency
- safer refactors
- more predictable theming
- clearer design intent
- less visual drift

Without tokens, systems usually accumulate arbitrary local choices.

That is where inconsistency begins.

---

## How tokens are used

Tokens are defined in CSS and then exposed through utilities.

```css
@theme inline {
  --radius-md: calc(var(--radius) - 2px);
  --shadow-sm: 0 1px 2px rgba(0,0,0,0.06);
}
```

Components consume those values indirectly:

```tsx
<Card className="rounded-lg shadow-sm bg-card border" />
```

The component does not need to know the raw value.

That is the point.

---

## Extending tokens safely

Not everything is tokenized yet.

That is intentional.

PyColors starts with high-impact foundations first.

As the system grows, you can extend tokens into categories such as:

- spacing
- motion
- z-index

The rule stays the same:

add tokens when the system clearly needs them, not preemptively.
