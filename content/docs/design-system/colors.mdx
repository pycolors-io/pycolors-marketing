# Colors define meaning before appearance

PyColors UI uses a **semantic color system** built to stay predictable across:

- light mode
- dark mode
- reusable components
- dashboards
- settings pages
- documentation
- marketing surfaces

The goal is simple:

> components should consume **color roles**, not invent visual values locally.

---

## Why semantic colors matter

Most UI inconsistency starts when components choose colors directly.

That creates problems like:

- hardcoded values
- theme drift
- weak contrast consistency
- duplicated visual intent
- surfaces that stop feeling related

PyColors avoids this by exposing **roles**, not raw colors.

That means components use utilities like:

- `bg-card`
- `text-foreground`
- `text-muted-foreground`
- `bg-primary`
- `border-border`

instead of raw values.

Semantic roles keep the system coherent as the product grows.

---

## Color architecture

PyColors uses an **OKLCH-based color system**.

OKLCH is useful because it is more perceptually predictable than raw RGB or HSL when you need:

- reliable contrast
- better light/dark consistency
- safer theme adjustments
- clearer control over visual balance

Core values are defined as CSS variables and then mapped into semantic Tailwind tokens.

---

## Theme mapping

The system maps CSS variables into Tailwind theme utilities.

  <Tab value="Code">

```css
@theme inline {
  --color-background: var(--background);
  --color-foreground: var(--foreground);
  --color-card: var(--card);
  --color-card-foreground: var(--card-foreground);
  --color-primary: var(--primary);
  --color-primary-foreground: var(--primary-foreground);
  --color-muted: var(--muted);
  --color-muted-foreground: var(--muted-foreground);
  /* ... */
}
```

  </Tab>
</Tabs>

This is what allows the whole UI to update coherently when the token layer changes.

## Semantic token roles

PyColors exposes high-level tokens that represent **meaning**, not decoration.

### Surfaces

- `--background`
- `--foreground`
- `--card`
- `--card-foreground`
- `--popover`
- `--popover-foreground`
- `--sidebar`
- `--sidebar-*`

These roles define how the interface is layered.

### Brand and component roles

- `--primary`
- `--primary-foreground`
- `--secondary`
- `--secondary-foreground`
- `--accent`
- `--accent-foreground`
- `--muted`
- `--muted-foreground`

These roles help distinguish:

- primary actions
- secondary emphasis
- subtle surfaces
- supporting UI

### State roles

- `--destructive`
- `--destructive-foreground`
- `--success`
- `--success-foreground`
- `--warning`
- `--warning-foreground`

These tokens communicate product meaning:

- danger
- success
- caution

They should remain consistent across all surfaces.

### Borders, inputs, and focus

- `--border`
- `--input`
- `--ring`

These roles keep form elements, containers, and focus states aligned with the rest of the system.

### Charts

- `--chart-1`
- `--chart-2`
- `--chart-3`
- `--chart-4`
- `--chart-5`

Chart tokens are intentionally separated from UI roles.

That prevents data visualization colors from leaking into interface components.

## Light and dark mode

PyColors uses CSS variables for both light and dark themes.

Light mode usually lives on `:root`.
Dark mode is applied with `.dark`.

```css
:root {
  --background: oklch(1 0 0);
  --foreground: oklch(0.141 0.005 285.823);

  --primary: oklch(0.21 0.006 285.885);
  --primary-foreground: oklch(0.985 0 0);
  /* ... */
}

.dark {
  --background: oklch(0.141 0.005 285.823);
  --foreground: oklch(0.985 0 0);

  --primary: oklch(0.92 0.004 286.32);
  --primary-foreground: oklch(0.21 0.006 285.885);
  /* ... */
}
```

The important idea is not the specific value.

It is the system behavior:

- components remain unchanged
- color meaning stays stable
- only the token layer changes underneath

## Role categories

| Role category | Tokens |
| --- | --- |
| Surfaces | `background`, `card`, `popover`, `sidebar` |
| Text | `foreground`, `muted-foreground`, `card-foreground` |
| Components | `primary`, `secondary`, `accent`, `muted` |
| States | `destructive`, `success`, `warning` |
| Inputs | `border`, `input`, `ring` |
| Data viz | `chart-1` → `chart-5` |

This table is useful because it clarifies intent before implementation.

---

## Variables reference

| Token | Purpose |
| --- | --- |
| `--background` | Base page surface |
| `--foreground` | Primary text |
| `--card` / `--card-foreground` | Card surfaces |
| `--primary` / `--primary-foreground` | Primary brand/action role |
| `--secondary` / `--secondary-foreground` | Secondary surfaces |
| `--muted` / `--muted-foreground` | Low-emphasis UI and text |
| `--accent` / `--accent-foreground` | Highlights and focus areas |
| `--destructive` / `--destructive-foreground` | Dangerous actions and states |
| `--success` / `--success-foreground` | Positive states |
| `--warning` / `--warning-foreground` | Caution states |
| `--border` | Borders and separators |
| `--input` | Form field surface |
| `--ring` | Focus ring |
| `--sidebar-*` | Sidebar theming |
| `--chart-1` → `--chart-5` | Chart palettes |

---

## Usage guidelines

- always use semantic utilities such as `bg-card` or `text-muted-foreground`
- never place raw OKLCH or hex values inside reusable components
- change tokens centrally, not per component
- keep state roles stable across the product
- preserve accessible contrast in both light and dark modes

If a repeated visual need appears, solve it in the token layer first.

---

## Common mistakes to avoid

- using raw color values inside components
- mixing semantic roles with one-off overrides
- treating state colors as decorative accents
- making each product area invent its own interpretation of “muted” or “accent”
- reusing chart colors for interface states

A design system becomes fragile when roles stop meaning the same thing everywhere.
