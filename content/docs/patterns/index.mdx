## Overview

Patterns are **reusable product-level blocks** built for real product work.

They go beyond isolated components and show **how SaaS features are actually assembled** across UI, behavior, state, and feedback.

<Callout title="Why Patterns exist">
  Patterns help you move faster with fewer UX mistakes by giving you opinionated,
  production-shaped feature structures instead of isolated primitives.
</Callout>

## Differences

| Section | Best for | Output |
| --- | --- | --- |
| **UI Components** | Single reusable primitives | Stateless, token-driven building blocks |
| **Guides** | Principles and reasoning | Explanations, composition rules, decision frameworks |
| **Patterns** | Real feature implementation | UI + behavior + state model + product UX |

### UI Components

- Single primitive
- Stateless
- Token-driven
- Reusable everywhere

### Guides

- Explanations and principles
- Composition rules
- “How to think” and avoid mistakes

### Patterns

- Feature-level building blocks
- UI + behavior + state model
- Real SaaS flows such as tables, overlays, and mutations
- Copy/paste implementations

## When to use

Use Patterns when:

- you’re building **real product features**, not just static screens
- you need **consistency across multiple pages**
- you want **predictable UX and feedback**
- accessibility and interaction rules matter

<Callout type="info" title="Good fit">
  Patterns are most useful when a feature has repeated structure, repeated state
  logic, or repeated UX decisions across the product.
</Callout>

## Common mistakes

- building features only with components, without a real product structure
- ignoring loading, empty, success, and error states
- making inconsistent UX decisions from page to page
- rebuilding the same feature flow multiple times

Patterns exist to eliminate these issues early.

## Why it matters

Without Patterns, teams often:

- rebuild the same logic multiple times
- introduce inconsistent UX
- slow down as the product grows
- create fragile feature implementations that are hard to scale

Patterns give you a **shared foundation** to ship faster with less friction.
